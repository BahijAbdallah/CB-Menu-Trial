Goal
Let admins control the DISPLAY ORDER of items inside each category, with NO frontend/design/image/content changes. 
Default order must be the current live order.

Do exactly this:

A) Storage (non-destructive)
- Add a single settings key: settings.itemOrderByCategory: Record<string, string[]>
  // e.g. { "salads": ["item_7","item_3","item_9"], "mains": [...] }
- Do NOT edit any item documents/rows.

B) One-time default seeding
- On server startup (or first access), if settings.itemOrderByCategory[catId] is missing:
  1) Fetch items for that category in their current DB/default order.
  2) Save their ids as the default array for that catId in settings.itemOrderByCategory.
- This preserves today’s order as the default without touching item data.

C) Admin panel (new page: Settings → Item Order)
- Dropdown to choose a category.
- Show a simple drag-and-drop list of that category’s items (title + tiny thumb, read-only).
- Buttons: “Save Order” and “Reset to Default”.
  • Save → PUT /api/settings/item-order { categoryId, order: string[] }
  • Reset → DELETE /api/settings/item-order?categoryId=... (removes entry; next read reseeds from current DB order)
- Do NOT modify any other fields; only write the array of IDs for that category.

D) API apply (so UI code stays unchanged)
- When serving items for a category (e.g., GET /api/items?categoryId=...):
  1) Load items in the usual way (unchanged query/filters).
  2) Load settings.itemOrderByCategory[catId] (seed if missing).
  3) Sort server-side before responding, using this stable helper:

     function sortBySavedOrder(items, catId, settings) {
       const order = settings?.itemOrderByCategory?.[catId] || [];
       const pos = new Map(order.map((id, i) => [id, i]));
       return items
         .map((it, idx) => ({ it, idx, p: pos.has(it.id) ? pos.get(it.id) : Number.POSITIVE_INFINITY }))
         .sort((a, b) => (a.p !== b.p ? a.p - b.p : a.idx - b.idx))
         .map(x => x.it);
     }

  4) Return the sorted array. 
- Important: Do NOT change any React components, markup, or CSS. The frontend keeps consuming the same endpoint; only the API response order changes.

E) Caching
- After admin “Save Order” or “Reset”, invalidate/revalidate any server cache for that category’s items so the new order appears on next load.

Acceptance
- Without any admin action, categories show the same order as today (seeded from current DB order).
- After admin reorders and saves, the frontend shows that exact order (1st, 2nd, 3rd…) on next fetch.
- No UI/design/images/content changes at all.
